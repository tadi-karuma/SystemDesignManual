## 1. はじめに：マニュアルの目的と対象

本マニュアルは、専門外の技術者・研究者・市民が、技術・制度・研究に関する構想を構造化・文書化し、AIの力を借りて社会に提案するまでの手順を体系的に示すものです。

👉 補助資料（倫理・構想者の心得）: [ETHICS.md](docs/ETHICS.md)

## 2. 構想とは何か：定義と構想者の立場

👉 構造的思考の基本や設計誤作動の分類については、[DESIGN_PRINCIPLES.md](docs/DESIGN_PRINCIPLES.md) を参照してください。

構想は、社会的・技術的な問いを「構造化」して差し出す行為です。
構想者は活動家・研究者・政策立案者と異なり、「設計と引き渡し」を担う中間者として振る舞います。
詳細は [ETHICS.md](docs/ETHICS.md) を参照してください。

## 3. 構想設計の全体フロー

構想は以下のような流れで進行します：
1. 動機とゴールの定義
2. 構造要素の逆算と整理
3. 設計図・文書化の作成（AI含む）
4. 提案と社会的公開
5. 反応と撤退条件までの設計

---

## 4. 動機とゴールの定義

### ✅ ステップ

* 解決したい課題、探究したい問い、創りたい未来像を明確にする
* そのゴールを定量的・定性的に定義する

### 💡 例

* 技術構想：気象衛星からのリアルタイムデータを都市防災に活用したい
* 制度構想：若者の政治参加を促す制度を設計したい
* 研究構想：低コストの再生可能エネルギー蓄積技術の有効性を検証したい

## 5. 逆算的構造化と設計思考

🔎 設計における誤作動パターンと健全な構造設計の原則は [DESIGN_PRINCIPLES.md](docs/DESIGN_PRINCIPLES.md) を参照してください。


### ✅ ステップ

* ゴールから逆算して必要な社会的・技術的インセンティブや条件をリスト化
* それを技術手段／制度／研究設計に具体化する

### 💡 ヒント

* 「人が行動したくなる／評価される／動機づけられる」仕組みに落とし込む
* 研究では「仮説を検証可能な形」にブレイクダウンする

## 7. AI活用と応答制御

### ✅ 推奨ツール

* ChatGPT / Claude / Gemini：構想案の整理・説明文書化
* Excalidraw / draw\.io / Mermaid：構造図の作成
* GitHub / Overleaf：文書群の構成・履歴管理

### ✅ AIへの指示テンプレート

```
目的：◯◯を達成するための構想を練りたい。
構想：□□という原理・方式に基づいて設計予定。
以下について明確に説明された文書を作って：
- 目的と動機
- 構造（制度・システム・手法）
- 期待される効果とリスク
```

---

### ⚠️ AI活用時のリスクと設計上の注意

#### ❌「答えているように見える」応答に騙される

* 見た目が整っていても、構文未読・文脈誤認が頻発する
* ✅ 対処：**必ず根拠（ファイル名・節名・構文抜粋）を要求する**

#### ❌「過去文脈を保持している」と信じすぎる

* GPTは会話履歴を曖昧に扱うため、**最新の構造だけを基準とする指示**が必須

#### ❌「正直に言う」前提で設計すると破綻する

* 確認していない情報を平然と「確認済み」と答えるリスクあり
* ✅ 対処：**「読めたと言うなら、出典やファイル構成を挙げよ」と明示的に問う**

---

### ✅ AI利用時のチェックリスト

#### 📘 基本確認

* [ ] 現行構成をAIが理解しているか？（ファイル構造で答えさせる）
* [ ] 回答の根拠がセクション名／文書名で明示されているか？

#### 🔎 構造読解時

* [ ] 「この構文を読んだか？」に対し構文単位で返せているか？
* [ ] 未読領域を補完していないか？

#### 🚫 誤動作対応

* [ ] 架空の要素や未確認情報が応答に混じっていたら、即時排除／修正したか？
* [ ] 虚偽・補完に基づいた出力があれば、後続文書に反映させたか？

#### 🧭 実装前の整合確認

* [ ] 構想全体が、どの根拠ファイル／読み取り結果に基づいて組まれているか？

## 8. 応答・対話の再整理フレーム

やり取りが10回以上に達した場合は、以下のテンプレートで文脈を再整理します：

```
# ▽要点まとめ
- 構想名：
- 現フェーズ：
- 現在の課題：
  ・課題1：
  ・課題2：
- 必要な支援：
  ・構造確認／仮説精査／類似事例調査
```

## 9. 引き渡し設計：提案と撤退条件

構想は「差し出す」ことを前提に設計されるべきです。引き渡し後の運用主体が変わることを想定し、
語りすぎず、構造を残して退く設計もまた構想の一部です。

---

## 10. 社会的反応の想定と対応方針

構想公開時には以下のような反応と誤解が生じる可能性があります：

- 意図しない誤読
- 過剰な期待と現実のギャップ
- 一部の否定的反応や敵対視

これらに対して構文・目的・制限を事前に明示し、撤退・保留の選択肢も設計に含めておきます。

## 13. よくある質問と標準回答テンプレ

構想提案時によく受ける質問と、それに対する標準回答テンプレ例：

**Q：なぜあなたがやるのか？**  
A：この構想は特定の主体でなくてもよく、構造提案として社会に開くためです。

**Q：この構想の実現性は？**  
A：実現可能性よりも「構造としての健全性」を検証・提示するのが第一段階です。
